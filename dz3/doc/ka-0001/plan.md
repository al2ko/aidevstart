# KA-0001. План реализации

## Уточняющие вопросы

1. Как пользователь вводит данные подключения: полный webhook URL Bitrix24 или отдельно адрес портала, ID пользователя и токен?
2. Чем считать проект для группировки: полем `GROUP_ID` задачи или другим пользовательским/CRM-признаком?
3. Нужно ли получать все страницы задач или достаточно первых 50 открытых задач?

## План

1. Исследовать заготовку внешней обработки `ЗадачиИзБитрикс24`: форму, реквизиты и текущий модуль формы.
2. Добавить на форму параметры подключения к Bitrix24 без хранения токена в коде и команду загрузки задач.
3. Реализовать серверный вызов REST-метода Bitrix24 `tasks.task.list` с фильтром по незакрытым задачам и явным `select` нужных полей.
4. Разобрать JSON-ответ, обработать ошибки подключения/API и преобразовать задачи в таблицу/дерево для вывода.
5. Вывести задачи в поле результата, сгруппировав их по проекту, и проверить сценарии: успешная загрузка, пустой список, ошибка токена/API.

## Изменяемые объекты и модули

- `ВнешняяОбработка.ЗадачиИзБитрикс24`
- Форма внешней обработки: `src/epf/ЗадачиИзБитрикс24/ЗадачиИзБитрикс24/Forms/Форма.xml`
- Расширение формы: `src/epf/ЗадачиИзБитрикс24/ЗадачиИзБитрикс24/Forms/Форма/Ext/Form.xml`
- Модуль формы: `src/epf/ЗадачиИзБитрикс24/ЗадачиИзБитрикс24/Forms/Форма/Ext/Form/Module.bsl`
- Модуль объекта внешней обработки - только если потребуется вынести общую серверную логику из модуля формы.
